# 📋 Полный аудит проекта Quiz Patente vs SKILL.md

## A. Executive Summary

**Текущее состояние:** Проект находится на финальной стадии Этапа 1 по плану разработки из SKILL.md. Все ключевые компоненты реализованы и работают на целевом устройстве (iPad Mini 2, iOS 12). Параллельно была выполнена значительная внеплановая работа: рефакторинг страницы ошибок (Errors Page), добавление SlideTransition, ConfirmationModal, useSwipe, useErrorTopics — всё это расширяет SKILL.md в правильном направлении. Финальный чеклист bug-analysis-and-fix-plan.md практически полностью закрыт (14 из 14 шагов с пометками DONE или выполнено).

**Ключевые проблемы:**
1. Этап 2 (StatsPage) отмечен в SKILL.md как следующий — но страница существует лишь как заглушка "В разработке". Компонент `TopicStatRow.jsx` не создан.
2. Несколько inline-стилей в `HomePage.jsx` и `CommentAccordion.jsx` не вынесены в CSS-классы (Шаг 11 bug-анализа помечен частично выполненным).
3. В `App.jsx` маршрут `/errors` не совпадает с тем, что описано в SKILL.md (`/quiz/errors`), — это архитектурное отклонение, хотя и осознанное (реализована отдельная страница выбора темы ошибок).
4. `.github/workflows/deploy.yml` упомянут в структуре SKILL.md, но файла нет в архиве (возможно, не включён в zip).

**Рекомендованный следующий шаг:** реализовать StatsPage с компонентом TopicStatRow, вынести оставшиеся inline-стили в CSS, и завести минимальную документацию по расхождениям маршрутизации.

## B. Структурированный разбор SKILL.md

Требования из SKILL.md, сгруппированные по блокам:

**Функциональные требования:**
- 25 тематических категорий с прогрессом на главной
- Квиз: 30 случайных вопросов, VERO/FALSO
- Пагинация: 30 кружков с цветовыми состояниями, кликабельная
- Перевод на русский: тогл, сохраняется на сессию
- Комментарий: открывается по кнопке после ответа
- Кнопка финиша: активна только когда все 30 отвечены
- ResultScreen: счёт, кнопки перезапуска/выхода/просмотра
- Страница ошибок (повтор неправильных вопросов)
- StatsPage с TopicStatRow
- DictionaryPage (Этап 3)

**Нефункциональные требования:**
- Совместимость с Chrome 92 / Safari iOS 12
- Нет `gap` в flexbox — использовать margins
- `loading="lazy"` + `alt=""` на всех изображениях
- Только CSS-переменные, никаких inline-стилей для статических значений
- Сервисный слой как единственная точка работы с данными
- `React.memo` на часто рендеримых компонентах
- Деплой на Vercel, CI/CD через GitHub

**Архитектурные требования:**
- `services/` → `hooks/` → `components/` — без обходов
- Данные только через `services/`, не импортировать JSON напрямую в компонентах
- localStorage только через `progressService` / `errorsService`
- Маршруты: `/`, `/quiz/:topicId`, `/stats`, `/dictionary`

## D. Пробелы и проблемы (по severity)

### 🔴 Critical

Нет critical-проблем. Все ранее критические баги (OOM, undefined CSS vars, borderBlock, антипаттерн useState) — исправлены согласно bug-analysis-and-fix-plan.md.

---

### 🟠 High

**H-1: StatsPage не реализована, TopicStatRow отсутствует**
Этап 2 в SKILL.md ("Статистика") формально является следующим после текущего. Страница существует как красивая заглушка, но функциональности ноль. `TopicStatRow.jsx` не создан вообще, хотя упомянут в структуре проекта. Пользователь видит "В разработке" — честно, но не полезно.

**H-2: Маршрутизация отличается от SKILL.md**
SKILL.md говорит: "Страница Errors отдельной страницей не нужна — это просто `/quiz/errors`". Реализован путь `/errors` с отдельной `ErrorsPage.jsx`. Это осознанное улучшение UX (выбор темы ошибок), но расхождение с SKILL.md не задокументировано. Если SKILL.md — живой документ, его нужно обновить.

---

### 🟡 Medium

**M-1: Inline-стили не полностью вынесены в CSS**
Шаг 11 bug-анализа помечен как частично выполненный: `ResultScreen.jsx` вынесен, но `HomePage.jsx` содержит 10+ inline-стилей (размеры изображений, отступы), `CommentAccordion.jsx` — 8+ inline-стилей для цветов и отступов. Это нарушает принцип SKILL.md: "Стили: только через CSS-классы, inline style только для динамических значений".

**M-2: useProgress.js использует старый var-стиль деструктуризации**
```js
var stateRef = useState(function () { return getProgress(); });
var progress = stateRef[0];
```
Bug-анализ исправил `useQuiz.js`, но `useProgress.js` и `useTopics.js` остались со старым антипаттерном. Это техдолг, который может путать при дальнейшей работе и мешает ESLint-хукам.

**M-3: QuizPagination — VERIFICA активна всегда, не только когда isFinished**
SKILL.md говорит: "Кнопка финиша (✅): Активна только когда все 30 вопросов отвечены (`isFinished === true`)". В реализации кнопка VERIFICA в пагинации всегда кликабельна — нет проверки `isFinished`. Пользователь может нажать её до завершения и получить ResultScreen с неполными результатами. `finish()` в `useQuiz` защищён `isSavedRef`, но UX страдает.

**M-4: Ключ `--spacing-*` используется, но в SKILL.md описана система `--space-*`**
SKILL.md документирует `--space-xs`, `--space-sm`, etc. В коде реализована система `--spacing-1..10`. Это не ошибка (система лучше), но SKILL.md устарел и вводит в заблуждение.

## F. Риски

**R-1 — Технический: отсутствие StatsPage блокирует полезность вкладки**
Пользователь видит иконку "Статистика" в BottomNav, нажимает, видит "В разработке". Это нормально на короткий срок, но снижает мотивацию пользоваться приложением регулярно. Без статистики сложно понять прогресс.

**R-2 — Технический: SKILL.md устаревает относительно реализации**
Несколько решений в коде уже отличаются от SKILL.md (маршрутизация, система спейсингов, расширения функциональности). Если не синхронизировать документ с кодом, он перестанет быть источником истины и следующая сессия работы начнётся с путаницы.

**R-3 — Качество кода: `useProgress.js` и `useTopics.js` — архаичный стиль**
Старый `var stateRef = useState(...)` стиль. Если ESLint-хуки включены, они будут давать ложные или пропущенные предупреждения. При рефакторинге этих файлов легко допустить ошибку.

**R-4 — UX: VERIFICA всегда кликабельна**
Пользователь может случайно нажать VERIFICA, не ответив на все вопросы. Результат будет показан с неполными данными. Не критично для личного использования, но нарушает контракт из SKILL.md.

**R-5 — Продуктовый: нет индикатора общего прогресса на главной**
HomePage показывает прогресс по каждой теме, но нет общего счётчика "отвечено X / 7095 вопросов". Для мотивации это было бы ценно, и данные уже доступны через `progressService`.

При переходе "назад" из квиза и повторном заходе в ту же тему снова идёт загрузка JSON. Простой кэш в `questionsService.js` (Map или sessionStorage) устранит повторные запросы. На iPad Mini 2 загрузка `topic_1.json` (675 КБ) занимает заметное время.

**I-2: Индикатор прогресса загрузки для режима "all"**
`loadAllQuestions` загружает 8.3 МБ пачками. Спиннер просто крутится — пользователь не знает, сколько осталось. Простой прогресс-индикатор (1/5, 2/5 батчей) улучшит UX.

**I-3: Offline-режим (PWA manifest + service worker)**
Приложение полностью статичное — идеальный кандидат для PWA. Service Worker закэширует JSON-файлы при первой загрузке, и квиз будет работать без интернета. Это особенно ценно для подготовки в дороге.

**I-4: Кнопка сброса прогресса в StatsPage**
Дать пользователю возможность сбросить всю статистику или статистику одной темы через `clearProgress()` / `clearErrors()`. Сейчас это возможно только через DevTools.
